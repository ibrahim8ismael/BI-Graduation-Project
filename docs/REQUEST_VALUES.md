# Request Values Reference

**Project:** Runner E-commerce ML API  
**Date:** 2026-06-13

---

## Request Flow

Every request follows this path:

```
Form Input → buildPayload() → useML hook → api.js → POST /api/proxy/<path> → FastAPI → ML Model
```

The frontend constructs a JSON payload from form fields, routes it through Next.js's server proxy (to avoid CORS), and forwards it to the FastAPI backend. The proxy is transparent — the payload arrives at the API unchanged.

## 1. Predict Category — `POST /predict/category`

**Route:** `/dashboard/product-manager/predict-category`  
**Role:** Product Manager  
**ML Model:** XGBoost Classifier (1000 trees, max_depth=16, lr=0.06)

### Request Fields

| Field | Type | Required | Source | In training data | Notes |
|---|---|---|---|---|---|
| `price` | `number` | Yes | User input | `price` column | Unit price of the product. Used directly by the model + to compute derived features. |
| `qty_ordered` | `integer` | Yes | User input | `qty_ordered` column | Quantity of units ordered. Used directly + to compute `revenue` and `qty_range`. |
| `value` | `number` | Auto-filled | Auto from `price × qty_ordered`, or manual override | `value` column | Original order value before discount. Auto-computed in the UI when user edits price or qty. If user edits this field directly, auto-fill disables. |
| `discount_amount` | `number` | Yes | User input | `discount_amount` column | Discount applied to the order. Also drives the `is_discounted` binary flag (1 if > 0). |
| `sku` | `string` | Yes | User input | `sku` column | Product SKU identifier. Parsed for prefixes to derive flags: `sku_contains_me`, `sku_contains_mat`, `sku_contains_wof`, `sku_contains_app`. |

### Auto-fill Behavior

When the user modifies `price` or `qty_ordered`, the UI auto-computes `value = price × qty_ordered`. If the user manually edits `value`, auto-fill is disabled until they clear it back to empty.

### Derived Features (computed by the ML pipeline, not sent in the request)

After receiving the payload, the API is expected to compute additional features that the model was trained on:

| Derived Feature | Formula | Purpose |
|---|---|---|
| `revenue` | `qty_ordered × price` | Total transaction value (also derivable from `value` after discount) |
| `discount_rate` | `discount_amount / value` | Discount proportion |
| `is_discounted` | `1 if discount_amount > 0 else 0` | Whether a discount was applied |
| `price_range` | Decile rank of `price` | Relative pricing tier |
| `qty_range` | Octile rank of `qty_ordered` | Relative order size |
| `sku_contains_*` | Substring match on `sku` | Product category hints in SKU naming convention |

### Training Data Source

The model was trained on `sales_06_FY2020-21 copy.csv`, a sales transactions CSV for fiscal year 2020-21. Features like `month` and `dayofweek` were extracted from `order_date` at training time but are not part of the request payload — the API is expected to handle them internally.

### Response

```json
{
  "predicted_category": "Electronics",
  "confidence": 0.92
}
```

| Field | Type | Description |
|---|---|---|
| `predicted_category` | `string` | Predicted product category label |
| `confidence` | `number` | Model confidence score (0–1) |

## 2. Segment Customer — `POST /segment/customer`

**Route:** `/dashboard/product-manager/segment-customer`  
**Role:** Product Manager  
**ML Model:** KMeans (k=4, trained on RFM features)

### Request Fields

| Field | Type | Required | Description |
|---|---|---|---|
| `recency` | `integer` | Yes | Days since the customer's **most recent order**. Lower = more recently active. |
| `frequency` | `integer` | Yes | **Total number of orders** placed by the customer over their lifetime. Higher = more engaged. |
| `monetary` | `number` | Yes | **Total revenue** generated by the customer (`sum` of `qty_ordered × price` across all orders). Higher = more valuable. |

### RFM Explained

**RFM** (Recency, Frequency, Monetary) is a behavioral segmentation framework used in customer analytics. It scores customers across three dimensions:

| Metric | What it measures | Low value → | High value → | Business meaning |
|---|---|---|---|---|
| **Recency** | Days since last purchase | Recently active ✅ | Churning/At risk ❌ | How engaged the customer is right now |
| **Frequency** | Total orders placed | Infrequent / one-off buyer | Loyal, repeat purchaser | How consistently they buy |
| **Monetary** | Total revenue generated | Low spender | High-value customer | How profitable the relationship is |

The model takes these 3 raw values, standardizes them (Z-score), and clusters them using **KMeans (k=4)** to assign a segment.

#### Segment Profiles

| Segment ID | Label | Recency | Frequency | Monetary | Profile |
|---|---|---|---|---|---|
| 1 | **VIP** | Very low (just ordered) | Very high | Very high | Best customers — most engaged |
| 0 | **Loyal** | Low (recent) | High | High | Solid repeat buyers |
| 3 | **Regular** | Moderate | Moderate | Moderate | Average — largest segment |
| 2 | **Lost** | High (long ago) | Low | Low | At risk / already churned |

#### Example

```
Recency=15, Frequency=100, Monetary=15000
→ Z-scale (recency=-0.8, frequency=2.1, monetary=1.9)
→ KMeans cluster 1 (closest centroid)
→ VIP segment
```

This customer ordered 15 days ago, placed 100 orders, and generated $15k in revenue. High across all dimensions → **VIP**.

### How the Model Uses These Values

1. The three values are **standardized** (Z-score via `StandardScaler`) so each feature has mean=0, variance=1.
2. The scaled values are fed into a **KMeans** model with 4 clusters.
3. Outliers where `monetary` is above the 99th percentile are dropped during training — the API may clip or reject such values.

### Segment Mapping

| Segment ID | Label | Typical Characteristics |
|---|---|---|
| 0 | **Loyal** | Frequent, recent buyers with moderate spend |
| 1 | **VIP** | High frequency, high monetary, very recent |
| 2 | **Lost** | Low recency (long time since last purchase), low frequency |
| 3 | **Regular** | Moderate values across all three metrics |

### Training Details

- **Aggregation**: Data grouped by `User Name`
- **Recency**: `(max order_date in dataset - customer's last order_date).days`
- **Frequency**: Count of `order_id` per customer
- **Monetary**: Sum of `revenue` per customer
- **Outlier removal**: Rows above 99th percentile of `Monetary` are excluded

### Response

```json
{
  "segment": "High-Value Active",
  "segment_id": 2,
  "characteristics": ["frequent_buyer", "high_spender", "recent"]
}
```

| Field | Type | Description |
|---|---|---|
| `segment` | `string` | Human-readable segment name |
| `segment_id` | `integer` | Numeric cluster ID (0–3) |
| `characteristics` | `array[string]` | Descriptors explaining why this segment was assigned |

## 3. Forecast Revenue — `POST /forecast/revenue`

**Route:** `/dashboard/ceo/forecast-revenue`  
**Role:** CEO  
**ML Model:** Prophet (yearly seasonality, additive, linear growth)

### Request Fields

| Field | Type | Required | Description |
|---|---|---|---|
| `periods` | `integer` | Yes | Number of forecast periods to generate. The training script uses 6 periods. Maps directly to Prophet's `periods` parameter — the number of time steps to forecast. |

### Training Details

- **Data**: Monthly sum of revenue (`qty_ordered × price`) per calendar month
- **Seasonality**: `yearly_seasonality=True` (models annual patterns), `weekly` and `daily` disabled
- **Growth**: `linear` (default) — Prophet fits piecewise linear trends with changepoints
- **Default forecast**: 6 months (`periods=6, freq='M'`)
- **Saved format**: Prophet model serialized to JSON (`prophet_revenue_model.json`)

### Notes

The API accepts a single `periods` integer. The training script hardcodes `periods=6` with monthly frequency. Sending a different value overrides the forecast horizon. The Prophet model uses yearly seasonality, so results may be more meaningful with a larger number of periods (e.g., 12 for a full year).

### Response

```json
{
  "forecast": [
    { "ds": "2021-10-03", "yhat": 5508534.22 },
    { "ds": "2021-10-10", "yhat": 3610075.4 },
    { "ds": "2021-10-17", "yhat": 1444654.68 }
  ]
}
```

| Field | Type | Description |
|---|---|---|
| `forecast` | `array` | Ordered list of forecast data points |
| `forecast[].ds` | `string` | Date in ISO format (YYYY-MM-DD) |
| `forecast[].yhat` | `number` | Predicted revenue value for that date |

## Common Behaviors

### Null Handling

When a number field is left empty, `buildPayload()` converts it to `null`:
```
"" (empty string) → null (JSON null)
```

### Type Coercion

- Number fields: `"123.45"` → `123.45`
- Text fields (sku): sent as-is
- NaN values: if parsing fails, the raw string is sent (may cause API validation errors)

### Request Paths (via Proxy)

| Endpoint | Proxy Path | FastAPI Path |
|---|---|---|
| Predict Category | `/api/proxy/predict/category` | `/predict/category` |
| Segment Customer | `/api/proxy/segment/customer` | `/segment/customer` |
| Forecast Revenue | `/api/proxy/forecast/revenue` | `/forecast/revenue` |

All requests go through the Next.js server proxy at `/api/proxy/[...segments]` which forwards them to the configured `NEXT_PUBLIC_FASTAPI_BASE_URL`.
